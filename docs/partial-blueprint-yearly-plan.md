# Partial Blueprint, Extended Yearly — Phase 1 Plan (Generative + Validation + Persistence)

This plan implements a **hybrid narrative architecture** where the LLM can generate story structure, but the engine remains the source of truth. The core idea is to store an initial **partial campaign blueprint** at new game creation, and then **extend it at the start of each in-game year** (or term) using a controlled pipeline: *generate → validate → persist → apply*.

The outcome is that each playthrough can have a truly distinct multi-year plot, while still being coherent, debuggable, and safe to save/load.

---

## 1) Why “partial blueprint + yearly expansion” is the right compromise

A single giant 7-year blueprint created up front is possible, but it tends to be brittle. You’ll either over-constrain creativity (making runs feel samey) or under-constrain it (making contradictions and pacing problems show up late).

Yearly expansion gives you three benefits:

1. **Better pacing control**: each year can escalate intentionally and respect what happened before.
2. **Reactive storytelling**: later-year arcs can be shaped by the player’s earlier choices (specialization, reputation, relationships, rule-breaking).
3. **Operational reliability**: smaller blueprint chunks are easier to validate and repair than one huge plan.

The tradeoff is that you must treat each year-plan as a **canonical artifact** stored in the save file.

---

## 2) High-level architecture

You’ll implement two closely related artifacts:

- **Campaign Blueprint (base)**: created at New Game. Defines long-lived “truths” and guardrails for this run.
- **Year Blueprint (year plan)**: generated at the start of each in-game year (1..7). Adds structured arcs, roles, clue ladders, and scheduled beats for that year.

Both are generated by the LLM as **structured JSON**, then validated by the engine, then persisted.

The engine uses these artifacts to:

- decide what plot modules are eligible
- place clues and events in locations that exist
- assign roles to NPCs that exist
- gate reveals and payoffs deterministically

The LLM is then used to narrate scenes consistent with the current blueprint + state.

## 3) Data model (Phase 1 friendly)

### 3.1 CampaignBlueprintBase (created once)

This should be small, stable, and mostly immutable. It sets the “physics of story” for the run.

Suggested fields:

- `blueprintVersion`: string
- `seed`: number or string
- `runThemes`: string[] (from allowlist)
- `toneProfile`: short tag(s) (from allowlist)
- `runTruths`: object of booleans/enums that describe the run’s special premise (from allowlist)
- `longLivedRoles`: a small mapping like mentor/rival/authority-contact assigned to NPC ids (optional in Phase 1 if NPC count is low)
- `yearPlans`: record of year → YearBlueprintSummary (just enough to know what’s been committed)

This base blueprint should not try to fully specify seven years. It should specify constraints and direction.

### 3.2 YearBlueprint (generated at year start)

A YearBlueprint defines the “season arc” structure and keeps the year coherent.

Phase 1 version can be quite modest:

- `year`: 1..7
- `arcModules`: 1–3 modules chosen from a registry (for Phase 1, start with 1 module)
- `moduleRoles`: per module, assign roles to NPC ids
- `clueSeeds`: list of clue IDs or clue slots with placement constraints (location IDs)
- `gates`: required flags/events before reveals
- `milestones`: 3–8 deterministic beats (“after first exam”, “after first library night”, etc.)
- `climax`: a small set of 2–3 possible climaxes with preconditions
- `compatibility`: a short list of constraints this year must respect (e.g., “no repeat of last year’s main module”) 

Later phases can deepen modules and ladders, but the architecture stays the same.

### 3.3 Canonical “Quest Flags” / “Facts”

Quest flags are still needed. They record what is now true, independent of narration.

In this design, flags are used for:

- year transitions (e.g. `year.1.completed`)
- module progression (e.g. `year.2.module.institutional_scandal.stage = 3`)
- reveal tracking (e.g. `clue.ashes_catalogue.found = true`)
- relationship milestones (e.g. `rel.npc.elias.milestone.shared_secret = true`)

Flags are not the plot. Flags are the *proof that plot progress happened.*

## 4) The yearly expansion pipeline

### 4.1 When expansion happens

Pick a deterministic trigger:

- **Start-of-year ceremony** (recommended): when the engine detects `year.N.completed` and time advances into the next year.

At that moment, you:

1. freeze and summarize relevant history
2. request a YearBlueprint from the LLM
3. validate it
4. persist it
5. apply the year’s initial state changes (unlock modules, seed clue slots, schedule beats)

### 4.2 Inputs to the generator

The YearBlueprint generation prompt should receive *only the information needed to design the next year*:

- `CampaignBlueprintBase` (themes/truths/tone)
- a compact **Year Recap** (engine-produced summary of major facts)
- current player build (specialization tags, reputation bands, key relationships)
- allowed content registries:
  - list of module IDs available
  - list of NPC IDs available for roles
  - list of location IDs available for clue placement
  - list of milestone templates available
- hard constraints:
  - must output JSON matching schema
  - must not introduce new IDs
  - must not contradict runTruths

### 4.3 Output: JSON-only YearBlueprint

Require the model to output only JSON. You then parse it.

### 4.4 Validation steps

Validation has two layers:

**A) Structural/schema validation**

- fields present
- correct types
- year number correct
- arrays within size limits

**B) Semantic validation**

- all referenced IDs exist in content registry
- no forbidden module combos
- clue placements only in existing locations
- milestones reference valid triggers
- climax options have satisfiable preconditions
- does not exceed year pacing constraints (e.g., not too many major arcs)

If invalid, run exactly one **repair pass**: feed back the validation errors and request corrected JSON.

If still invalid, fall back to a **deterministic default YearBlueprint** (“safe year”) so the game never bricks.

### 4.5 Persistence

Persist the validated YearBlueprint immediately:

- store it under `GameState.blueprint.yearPlans[year]`
- store a `yearPlanHash` or `yearPlanId` for debugging
- store a `promptVersion` so future changes are traceable

This makes the year plan reproducible forever.

## 5) How the engine uses blueprints during play

The blueprint is not “narration.” It is an eligibility and gating layer.

During action resolution, the engine can consult the current year plan to decide:

- what modules are active
- what clue slots are eligible this week/term
- what reveals are permitted given flags
- what special events can appear in the scheduler

The important principle is:

- The engine decides *what* is revealed and when.
- The LLM decides *how it feels* and how it’s described.

### Example

Player explores the library.

Engine:

- checks current year plan module(s)
- checks if any clue slots are eligible
- chooses a clue deterministically (seeded RNG)
- sets flags: `clue.X.found = true`, `module.stage = 2`

LLM:

- describes the library scene
- describes the clue text in an atmospheric way
- offers dialogue options consistent with present NPCs

## 6) Phase 1 scope: what to implement vs postpone

### Implement in Phase 1

- CampaignBlueprintBase structure and persistence
- One YearBlueprint generator + validator
- Year transition trigger and persistence
- A minimal module registry (even if it’s only 1–2 modules)
- A minimal milestone system (even if milestones are simple)
- A compact recap builder (non-LLM is fine at first)
- LLM prompt templates for blueprint generation (JSON output)
- Repair pass and deterministic fallback

### Postpone to Phase 2/3

- multi-module year combinatorics
- deep reveal ladders with many stages
- rich scheduler integration across dozens of events
- sophisticated content pack override rules
- full plugin API

This keeps Phase 1 achievable while still proving the hybrid design.

---

## 7) Recommended “blueprint boundaries” (what the LLM may and may not do)

To keep coherence strong, enforce these rules in code:

- The LLM may define **which module** runs this year (from allowlist) and assign **roles** (from NPC id list).
- The LLM may propose **milestones** using your milestone templates.
- The LLM may place **clue slots** only in known locations.
- The LLM may not invent new NPCs, spells, or locations.
- The LLM may not declare outcomes as facts (“the professor is guilty”) unless the blueprint schema has a hidden-truth field and you accept it.

In other words, the LLM is the architect inside a building code.

---

## 8) Minimum viable yearly module library (so years don’t feel repetitive)

Even with a small content base, you can make years distinct by varying module themes.

A starter set of modules that are *not* “bad guy hurts somebody”:

- **Institutional Scandal**: coverups, evidence chains, hearings, reputations.
- **Forbidden Method Debate**: ethics of a technique, peer pressure, policy fights.
- **Metaphysics Leak**: anomalies that escalate, competing interpretations.
- **Resource Crisis**: budget, access, rationing of rare reagents, black markets.
- **Faction Constitution**: elections, reforms, curricular control, alliances.
- **Academic Prestige War**: credit stealing, sabotage, publication-like rivalry.

Each module can have a reveal ladder and 2–3 climax shapes.

Yearly planning then becomes “choose one module + role assignment + reveal pacing.”

## 9) Practical implementation sketch (use-case driven)

### Use cases you’ll add in Phase 1

- `StartNewGame`
  - generates CampaignBlueprintBase (optionally via LLM) OR uses a safe default
  - persists state

- `GenerateYearBlueprint(year)`
  - builds recap + registry lists
  - calls LLM
  - validates + repairs + persists

- `AdvanceToNextYear`
  - sets `year.N.completed`
  - triggers `GenerateYearBlueprint(N+1)`
  - applies year-start effects (unlock modules, seed clue slots)

- `PerformAction`
  - resolves action
  - consults current year plan for eligibility
  - updates flags
  - builds narration context

---

## 10) Definition of done for Phase 1 (with yearly expansion)

Phase 1 is complete when you can:

- start a new game and store a base blueprint
- play through a small set of actions in Year 1
- reach a year transition trigger
- generate a Year 2 blueprint via LLM
- validate and persist it
- continue play in Year 2 with at least one visibly different arc module or milestone path
- save and load with the blueprint intact

At that point you’ve proven the full hybrid architecture while keeping the implementation small.
